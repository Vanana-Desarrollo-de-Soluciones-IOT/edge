# Historias Técnicas Unificadas — Clair IoT (Edge Service)

Este documento centraliza todas las Épicas e Historias Técnicas correspondientes al servicio de borde (Edge Service) de la plataforma Clair IoT.

---

## Epics

| Epic / Story ID | Título | Descripción | Criterios de Aceptación | Relacionado con (Epic ID) |
| :--- | :--- | :--- | :--- | :--- |
| ES-EP-01 | Gestión de Identidad y Acceso de Dispositivos | Como Usuario, quiero gestionar la autenticación y el estado de presencia de los dispositivos edge, para asegurar que solo los equipos autorizados puedan enviar datos y poder monitorear su estado de actividad en tiempo real. | La Épica se completa cuando todas las historias de usuario relacionadas han sido implementadas, validadas y aceptadas. | - |
| ES-EP-02 | Gestión de Aprovisionamiento de Dispositivos | Como Usuario, quiero gestionar la sincronización y el aprovisionamiento de dispositivos en tiempo real desde el sistema central, para que el caché de dispositivos locales esté actualizado y permita la autenticación correcta en el edge. | La Épica se completa cuando todas las historias de usuario relacionadas han sido implementadas, validadas y aceptadas. | - |
| ES-EP-03 | Gestión de Telemetría de Dispositivos | Como Usuario, quiero gestionar la ingesta de telemetría y el monitoreo de estado de los dispositivos, para que la plataforma pueda recopilar datos ambientales y rastrear la disponibilidad de los equipos. | La Épica se completa cuando todas las historias de usuario relacionadas han sido implementadas, validadas y aceptadas. | - |
| ES-EP-04 | Control de Comandos de Dispositivos | Como Usuario, quiero gestionar el envío y confirmación de comandos de dispositivos, para que la plataforma pueda controlar el hardware remoto y rastrear el estado de ejecución de los comandos. | La Épica se completa cuando todas las historias de usuario relacionadas han sido implementadas, validadas y aceptadas. | - |

## Historias Técnicas (Technical Stories)

| Epic / Story ID | Título | Descripción | Criterios de Aceptación | Relacionado con (Epic ID) |
| :--- | :--- | :--- | :--- | :--- |
| ES-US-01 | Autenticación de Credenciales de Dispositivo | Como Desarrollador, quiero que el sistema valide el identificador de hardware y la clave de API del dispositivo físico, para que la autenticación esté disponible al construir solicitudes seguras para mis aplicaciones. | **Escenario: Autenticación exitosa del dispositivo**<br>Dado que el servicio de autenticación local está activo<br>Cuando se realiza una solicitud de API con cabeceras "X-Hardware-Id" y "X-API-Key" válidas y registradas<br>Entonces el sistema valida y autoriza la solicitud de datos.<br><br>**Escenario: Rechazo de solicitud por credenciales faltantes**<br>Dado que el servicio de autenticación local está activo<br>Cuando se realiza una solicitud de API y falta la cabecera "X-Hardware-Id" o "X-API-Key"<br>Entonces el sistema rechaza la solicitud devolviendo un código de estado 401<br>Y retorna el mensaje de error "Missing X-Hardware-Id or X-API-Key".<br><br>**Escenario: Rechazo de solicitud por credenciales inválidas**<br>Dado que el servicio de autenticación local está activo<br>Cuando se realiza una solicitud de API con credenciales que no coinciden con la base de datos local<br>Entonces el sistema rechaza la solicitud devolviendo un código de estado 401<br>Y retorna el mensaje de error "Invalid hardware ID or API key". | ES-EP-01 |
| ES-US-02 | Monitoreo de Presencia y Transición de Estado a Fuera de Línea | Como Desarrollador, quiero que el sistema identifique dispositivos inactivos en segundo plano y publique sus cambios de estado, para que los eventos de conectividad actualizados estén disponibles para mis aplicaciones. | **Escenario: Dispositivo inactivo marcado como OFFLINE**<br>Dado que el monitor de presencia ("DevicePresenceMonitor") se ejecuta periódicamente cada 5 segundos<br>Cuando un dispositivo no ha enviado datos o registrado actividad en los últimos 30 segundos<br>Entonces el sistema cambia su estado interno de presencia a "OFFLINE" en la base de datos local.<br><br>**Escenario: Publicación de cambio de presencia en Kafka**<br>Dado que se detecta una transición de estado de presencia (ONLINE a OFFLINE o viceversa) de un dispositivo<br>Cuando se completa la actualización en la base de datos local<br>Entonces el sistema publica un evento de integración "DevicePresenceChangedEvent" en el tema de Kafka "clair.iam.devices.presence" con los campos device_id, hardware_id, status y ocurrió_at. | ES-EP-01 |
| ES-US-03 | Sincronizar Evento de Aprovisionamiento de Dispositivos | Como Desarrollador, quiero que el sistema consuma y procese eventos de cambio de dispositivos desde un tema de Kafka, para que la información del dispositivo esté disponible para mis aplicaciones en el caché local. | **Escenario: Procesar evento de aprovisionamiento con datos válidos**<br>Dado que el consumidor de Kafka para "clair.provisioning.devices.changed" está activo<br>Cuando se recibe un mensaje con los campos device_id, hardware_id, api_key y status válidos<br>Entonces el sistema normaliza el payload al esquema local<br>Y actualiza o inserta el registro del dispositivo en la tabla de caché local "devices" con los valores correspondientes.<br><br>**Escenario: Rechazar procesamiento del evento por campos obligatorios faltantes**<br>Dado que el consumidor de Kafka para "clair.provisioning.devices.changed" está activo<br>Cuando se recibe un mensaje que carece de alguno de los campos requeridos (device_id, hardware_id, api_key o status)<br>Entonces el sistema lanza un error de validación (ValueError)<br>Y descarta el mensaje sin guardar cambios en la base de datos local.<br><br>**Escenario: Soportar formatos camelCase y snake_case en el payload**<br>Dado que el consumidor de Kafka para "clair.provisioning.devices.changed" está activo<br>Cuando se recibe un mensaje en formato camelCase (deviceId, hardwareId, apiKey) o snake_case (device_id, hardware_id, api_key)<br>Entonces el sistema normaliza las claves correctamente al formato local<br>Y realiza el registro en la base de datos local de manera exitosa. | ES-EP-02 |
| ES-US-04 | Ingestar Telemetría de Dispositivos | Como Desarrollador, quiero enviar datos de telemetría a través de la API REST, para que estén disponibles al construir funcionalidades en mis aplicaciones. | **Escenario: Ingesta de telemetría exitosa**<br>Dado que el endpoint "/api/v1/device/telemetry" está disponible<br>Y la solicitud está autenticada con cabeceras X-Hardware-Id y X-API-Key válidas<br>Cuando se envía una solicitud POST con valores para deviceId, timestamp, uptime, airQuality (co2, temperature, humidity), particulateMatter (pm1_0, pm2_5, pm10), connectivity (status, network, signalStrength), location (country), healthStatus y status<br>Entonces se recibe una respuesta con Estado 201<br>Y se incluye el registro de telemetría en el cuerpo de la respuesta, con un ID único, marca de tiempo recorded_at y los valores enviados.<br><br>**Escenario: Rechazar telemetría por falta de autenticación**<br>Dado que el endpoint "/api/v1/device/telemetry" está disponible<br>Cuando se envía una solicitud POST con un cuerpo JSON válido pero faltan las cabeceras X-Hardware-Id o X-API-Key<br>Entonces se recibe una respuesta con Estado 401<br>Y se incluye un mensaje de error "Missing X-Hardware-Id or X-API-Key" en el cuerpo de la respuesta.<br><br>**Escenario: Rechazar telemetría por credenciales inválidas**<br>Dado que el endpoint "/api/v1/device/telemetry" está disponible<br>Cuando se envía una solicitud POST con cabeceras X-Hardware-Id o X-API-Key inválidas<br>Entonces se recibe una respuesta con Estado 401<br>Y se incluye un mensaje de error "Invalid hardware ID or API key" en el cuerpo de la respuesta.<br><br>**Escenario: Rechazar telemetría por campos obligatorios faltantes**<br>Dado que el endpoint "/api/v1/device/telemetry" está disponible<br>Y la solicitud está autenticada con cabeceras válidas<br>Cuando se envía una solicitud POST pero faltan campos obligatorios como deviceId<br>Entonces se recibe una respuesta con Estado 400<br>Y se incluye un mensaje de error que indica el campo faltante en el cuerpo de la respuesta. | ES-EP-03 |
| ES-US-05 | Obtener Comandos Pendientes | Como Desarrollador, quiero obtener la lista de comandos pendientes de un dispositivo a través de la API REST, para que el dispositivo edge pueda consultar las acciones que debe ejecutar. | **Escenario: Obtener comandos pendientes de forma exitosa**<br>Dado que el endpoint "/api/v1/device/commands/pending" está disponible<br>Y la solicitud está autenticada con cabeceras X-Hardware-Id y X-API-Key válidas<br>Cuando se envía una solicitud GET<br>Entonces se recibe una respuesta con Estado 200<br>Y se devuelve un cuerpo JSON con el conteo (count) y la lista de comandos con los campos commandId, deviceId, hardwareId, type, status, payload, receivedAt, deliveredAt, acknowledgedAt y failureReason.<br><br>**Escenario: Rechazar obtención de comandos por falla de autenticación**<br>Dado que el endpoint "/api/v1/device/commands/pending" está disponible<br>Cuando se envía una solicitud GET con credenciales faltantes o inválidas<br>Entonces se recibe una respuesta con Estado 401. | ES-EP-04 |
| ES-US-06 | Confirmar Ejecución de Comando | Como Desarrollador, quiero enviar una confirmación de ejecución de comando a través de la API REST, para que el estado de ejecución quede registrado y disponible para las aplicaciones. | **Escenario: Confirmar ejecución de comando de forma exitosa con estado EXECUTED**<br>Dado que el endpoint "/api/v1/device/commands/{command_id}/ack" está disponible<br>Y la solicitud está autenticada con cabeceras X-Hardware-Id y X-API-Key válidas<br>Cuando se envía una solicitud POST con el estado "EXECUTED"<br>Entonces se recibe una respuesta con Estado 200<br>Y se devuelven los detalles del comando actualizado con estado "EXECUTED" y la marca de tiempo acknowledgedAt.<br><br>**Escenario: Confirmar ejecución de comando con estado FAILED y motivo de falla**<br>Dado que el endpoint "/api/v1/device/commands/{command_id}/ack" está disponible<br>Y la solicitud está autenticada con cabeceras válidas<br>Cuando se envía una solicitud POST con el estado "FAILED" y el motivo de falla failureReason "Embedded timeout"<br>Entonces se recibe una respuesta con Estado 200<br>Y se devuelven los detalles del comando actualizado con estado "FAILED", el motivo "Embedded timeout" y la marca de tiempo acknowledgedAt.<br><br>**Escenario: Rechazar confirmación por valor de estado inválido**<br>Dado que el endpoint "/api/v1/device/commands/{command_id}/ack" está disponible<br>Y la solicitud está autenticada con cabeceras válidas<br>Cuando se envía una solicitud POST con un valor de estado distinto de EXECUTED o FAILED<br>Entonces se recibe una respuesta con Estado 400<br>Y se incluye el mensaje de error "status must be EXECUTED or FAILED" en el cuerpo de la respuesta. | ES-EP-04 |
| ES-US-07 | Consultar Estado de Conexión del Dispositivo | Como Desarrollador, quiero consultar el estado de conexión de un dispositivo a través de la API REST, para verificar si un equipo está en línea o fuera de línea al construir paneles de control. | **Escenario: Obtener estado de conexión de un dispositivo activo**<br>Dado que el endpoint "/api/v1/device/{hardware_id}/connection-status" está disponible<br>Cuando se envía una solicitud GET para un hardware_id existente<br>Entonces se recibe una respuesta con Estado 200<br>Y el cuerpo de la respuesta incluye hardware_id, status (ONLINE u OFFLINE), la marca de tiempo last_seen_at y seconds_since_last_seen.<br><br>**Escenario: Intentar obtener estado de conexión de un dispositivo inexistente**<br>Dado que el endpoint "/api/v1/device/{hardware_id}/connection-status" está disponible<br>Cuando se envía una solicitud GET con un hardware_id que no existe en el sistema<br>Entonces se recibe una respuesta con Estado 404<br>Y se devuelve un mensaje de error indicando que el dispositivo no fue encontrado. | ES-EP-03 |
