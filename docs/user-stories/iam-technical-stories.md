# Historias de Usuario - IAM (Identity and Access Management)

Esta sección presenta las Épicas y las Historias Técnicas (Technical Stories) identificadas para el bounded context de control de accesos e identidad (`iam`). Cada historia está redactada desde la perspectiva de un rol de la plataforma (las Historias Técnicas usan al Desarrollador para habilitar capacidades técnicas) e incluye criterios de aceptación detallados mediante escenarios con el formato Dado que / Cuando / Entonces.

## Épicas

| Epic / Story ID | Título | Descripción | Criterios de Aceptación | Relacionado con (Epic ID) |
| --------------- | ------ | ----------- | ----------------------- | ------------------------- |
| ES-EP-01 | Gestión de Identidad y Acceso de Dispositivos | Como Usuario, quiero gestionar la autenticación y el estado de presencia de los dispositivos edge, para asegurar que solo los equipos autorizados puedan enviar datos y poder monitorear su estado de actividad en tiempo real. | La Épica se completa cuando todas las historias de usuario relacionadas han sido implementadas, validadas y aceptadas. | - |

## Historias Técnicas (Technical Stories)

| Epic / Story ID | Título | Descripción | Criterios de Aceptación | Relacionado con (Epic ID) |
| --------------- | ------ | ----------- | ----------------------- | ------------------------- |
| ES-US-01 | Autenticación de Credenciales de Dispositivo | Como Desarrollador, quiero que el sistema valide el identificador de hardware y la clave de API del dispositivo físico, para que la autenticación esté disponible al construir solicitudes seguras para mis aplicaciones. | **Escenario: Autenticación exitosa del dispositivo**<br>Dado que el servicio de autenticación local está activo<br>Cuando se realiza una solicitud de API con cabeceras "X-Hardware-Id" y "X-API-Key" válidas y registradas<br>Entonces el sistema valida y autoriza la solicitud de datos.<br><br>**Escenario: Rechazo de solicitud por credenciales faltantes**<br>Dado que el servicio de autenticación local está activo<br>Cuando se realiza una solicitud de API y falta la cabecera "X-Hardware-Id" o "X-API-Key"<br>Entonces el sistema rechaza la solicitud devolviendo un código de estado 401<br>Y retorna el mensaje de error "Missing X-Hardware-Id or X-API-Key".<br><br>**Escenario: Rechazo de solicitud por credenciales inválidas**<br>Dado que el servicio de autenticación local está activo<br>Cuando se realiza una solicitud de API con credenciales que no coinciden con la base de datos local<br>Entonces el sistema rechaza la solicitud devolviendo un código de estado 401<br>Y retorna el mensaje de error "Invalid hardware ID or API key". | ES-EP-01 |
| ES-US-02 | Monitoreo de Presencia y Transición de Estado a Fuera de Línea | Como Desarrollador, quiero que el sistema identifique dispositivos inactivos en segundo plano y publique sus cambios de estado, para que los eventos de conectividad actualizados estén disponibles para mis aplicaciones. | **Escenario: Dispositivo inactivo marcado como OFFLINE**<br>Dado que el monitor de presencia ("DevicePresenceMonitor") se ejecuta periódicamente cada 5 segundos<br>Cuando un dispositivo no ha enviado datos o registrado actividad en los últimos 30 segundos<br>Entonces el sistema cambia su estado interno de presencia a "OFFLINE" en la base de datos local.<br><br>**Escenario: Publicación de cambio de presencia en Kafka**<br>Dado que se detecta una transición de estado de presencia (ONLINE a OFFLINE o viceversa) de un dispositivo<br>Cuando se completa la actualización en la base de datos local<br>Entonces el sistema publica un evento de integración "DevicePresenceChangedEvent" en el tema de Kafka "clair.iam.devices.presence" con los campos device_id, hardware_id, status y ocurrió_at. | ES-EP-01 |
